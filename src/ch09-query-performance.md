# Chapter 9: Query Performance and the Optimizer

We've built the machine (Part I), the engine (Part II), and the pipeline (Part III). Now we use them. Query performance is where all those layers meet: the cost-based optimizer chooses access methods based on statistics stored in persistent InnoDB tables; the chosen plan touches pages that may reside in the buffer pool or require storage I/O; locks acquired during execution interact with the transaction system's MVCC and undo logs; and the redo log records generated by writes flow through the async commit architecture to storage and then to replicas. Every query is a cross-section of the entire system.

This chapter brings that cross-section into focus. We cover reading execution plans correctly, maintaining plan stability under Aurora MySQL's constraints, interpreting Aurora-specific wait events, and eliminating the anti-patterns that dominate production incident post-mortems. By the end, you will be able to trace a slow query from its EXPLAIN plan through the storage and replication layers, identifying the precise bottleneck at each stage.

## 9.1 Reading Execution Plans

### EXPLAIN Formats: When to Use Each

Aurora MySQL 3.x supports three `EXPLAIN` output formats. The **traditional tabular format** (default) shows one row per table in join order with columns `id`, `select_type`, `table`, `type`, `possible_keys`, `key`, `key_len`, `rows`, `filtered`, and `Extra`. Use this for rapid assessment — the `type` column reveals the access method, `rows` shows the optimizer's estimate, and `Extra` signals covering indexes (`Using index`), filesorts (`Using filesort`), or temporary tables (`Using temporary`). The traditional format omits cost breakdowns and nests subquery plans poorly, so reach for alternatives when plans exceed three tables.

`EXPLAIN FORMAT=JSON` outputs structured cost information including `query_cost`, `read_cost`, `eval_cost`, and `prefix_cost` for each table access, plus nested subquery detail [^457^]. Use JSON format when comparing two candidate plans quantitatively or feeding plan data into monitoring tools.

The **TREE format**, introduced in MySQL 8.0.16, mirrors the iterator-based execution engine and should be your default for complex joins [^457^]. Leaf nodes represent data access; parent nodes represent operations. Read from the inside out — the root node shows total query cost. TREE format makes join order and filter pushdown visible in a way that tabular output obscures.

For comparing estimates against reality, use `EXPLAIN ANALYZE` (MySQL 8.0.18+). It executes the query and reports actual timing alongside estimates [^457^]:

```sql
-- actual time=X..Y: ms to first row and all rows; rows=N: actual rows returned
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42 AND status = 'shipped';
```

When estimated rows diverge from actual by more than 3x, the optimizer will likely choose suboptimal join orders and access methods. For example, if `EXPLAIN` estimates 894 rows but `EXPLAIN ANALYZE` reveals 2,844 actual rows on a filter step, the statistics are stale and `ANALYZE TABLE` is warranted. For DML statements, wrap in a transaction and roll back:

```sql
START TRANSACTION;
EXPLAIN ANALYZE UPDATE orders SET status = 'processed' WHERE order_date < '2024-01-01';
ROLLBACK;
```

### Access Methods: From Const to ALL

The `type` column indicates row access strategy. The hierarchy from best to worst is: `system` > `const` > `eq_ref` > `ref` > `index_merge` > `range` > `index` > `ALL` [^8^].

| Access Method | Trigger Condition | Cost on 1M Row Table | Production Signal |
|:---|:---|:---|:---|
| `const` | PK or unique index equality: `WHERE id = 42` | 1 row examined | Ideal for point lookups |
| `eq_ref` | Join on PK/unique NOT NULL: `JOIN tbl ON fk = pk` | 1 row per outer row | Best possible join type |
| `ref` | Non-unique index equality: `WHERE idx_col = 'val'` | Rows matching value | OK when selectivity > 1% |
| `range` | Index range scan: `WHERE col BETWEEN a AND b` | O(log n + matches) | Expected for dates, pagination |
| `index` | Full index tree scan (covering or ordered) | O(n) index entries | OK only with `Using index` in Extra |
| `ALL` | Full table scan, no usable index | O(n) rows | Unacceptable on tables > 100K rows [^8^] |

`const` is chosen when at most one row matches via a primary key or unique index equality. `eq_ref` is the best join type — one inner row per outer row via a primary key or unique NOT NULL index. `ref` reads all rows matching a non-unique index value; efficient when cardinality is high, but degenerating toward a scan when many rows share the same value. `range` applies when a key column uses `=`, `<>`, `>`, `>=`, `<`, `<=`, `IS NULL`, `BETWEEN`, `LIKE` (no leading wildcard), or `IN()` [^8^]. The `index` type indicates a full index tree scan — acceptable only when `Using index` appears in `Extra`, signaling a covering index that eliminates base table access. Without `Using index`, treat `index` as a scan in disguise.

`ALL` is the full table scan. On Aurora, this drives `VolumeReadIOPS` upward, directly affecting I/O costs. When `EXPLAIN` shows `ALL` on a table exceeding 100,000 rows, investigate: add an index, rewrite the query, or confirm the table is small enough to scan. Remember from Chapter 3: every page read from storage must traverse the network to the distributed storage layer, making full table scans far more expensive on Aurora than on local-disk MySQL.

### Join Algorithms: Nested Loop, Hash Join, and Batched Key Access

The **nested loop join (NLJ)** is MySQL's default. For each outer table row, the engine scans the inner table — O(N * M) without indexes, O(N * log(M)) with an index on the inner table. The optimizer chooses the smaller table as the outer loop when statistics are accurate; stale stats can reverse this and cause catastrophic performance.

**Hash join** (MySQL 8.0.18+) replaces Block Nested Loop for equi-joins without suitable indexes [^8^]. It builds an in-memory hash table from the smaller input, then probes with the larger. Hash join uses `xxHash64`, can spill to disk across 128 chunk files, and is controlled by `join_buffer_size`. It does not guarantee output ordering — `ORDER BY` requires a separate sort. From MySQL 8.0.20 onward, BNL is no longer used at all [^8^].

```sql
-- Control hash join via optimizer_switch (block_nested_loop controls it in 8.0.19+)
SET optimizer_switch = 'block_nested_loop=on';    -- Enable
SELECT /*+ NO_BNL(t1, t2) */ * FROM t1 JOIN t2 ON t1.a = t2.a;  -- Disable per-query
```

**Batched Key Access (BKA)** accumulates rows from the first operand in a join buffer, then performs index lookups in batch through the MRR interface, converting random I/O to sequential access [^8^]. Enable with `SET optimizer_switch='mrr=on,mrr_cost_based=off,batched_key_access=on'`. BKA requires all three flags — `mrr_cost_based=off` because the cost model is too pessimistic. Verify with `Using join buffer (Batched Key Access)` in `EXPLAIN`.

## 9.2 Statistics and Plan Stability

### Persistent Statistics and When to Refresh Them

In Aurora MySQL 3.x, persistent optimizer statistics are enabled by default (`innodb_stats_persistent=ON`) and survive restarts [^497^]. Statistics live in `mysql.innodb_table_stats` and `mysql.innodb_index_stats`. Auto-recalculation triggers when more than 10% of rows change (`innodb_stats_auto_recalc=ON`) [^497^], but this threshold can be too conservative — a gradual 5% drift may leave plans suboptimal without crossing it.

Run `ANALYZE TABLE` manually after bulk loads, large `DELETE` operations, when `EXPLAIN` estimates diverge from actual by 3x, and after index creation [^497^]:

```sql
-- Update statistics and histograms
ANALYZE TABLE orders UPDATE HISTOGRAM ON status, customer_type;

-- Check last update time
SELECT table_name, last_update, n_rows
FROM mysql.innodb_table_stats
WHERE table_name = 'orders';
```

The `innodb_stats_persistent_sample_pages` parameter (default 20) controls index pages sampled. Increase to 100-500 for large tables with skewed distributions — more accurate cardinality estimates at the cost of longer `ANALYZE TABLE` execution [^497^].

### Histograms: Implementation and Impact

MySQL 8.0 supports histogram statistics for non-indexed columns with 1-1,024 buckets, providing distribution information that improves selectivity estimates beyond the uniform-distribution assumption [^497^]. Histograms are most valuable for skewed columns — status fields with 90% "completed" and 10% "pending", or customer tiers with long tails.

```sql
-- Create histogram with auto-update
ANALYZE TABLE orders UPDATE HISTOGRAM ON status WITH 200 BUCKETS AUTO UPDATE;

-- View histogram data
SELECT * FROM INFORMATION_SCHEMA.COLUMN_STATISTICS
WHERE TABLE_NAME = 'orders' AND COLUMN_NAME = 'status';
```

The `condition_fanout_filter` flag of `optimizer_switch` controls whether histograms are consulted during cardinality estimation [^497^]. Histograms are updated only on `ANALYZE TABLE` and impose no write-time overhead — the tradeoff versus indexes is that indexes accelerate both filtering and retrieval but cost maintenance on every write. For frequently filtered columns, a composite index is usually the better investment; use histograms for columns that need distribution awareness without index maintenance cost.

### The Critical Limitation: No Query Plan Management

Critically, Aurora's **Query Plan Management (QPM)** — recording, evaluating, and enforcing known good plans — is available only for Aurora PostgreSQL via `apg_plan_mgmt`, not for Aurora MySQL [^QPM^]. This is one of the most significant capability gaps between the two engines. Aurora PostgreSQL DBAs can capture baseline plans, detect regressions automatically, and force approved plans when the optimizer chooses poorly. Aurora MySQL DBAs have none of this.

Plan stability must be achieved through `optimizer_switch` flags, index hints (`USE INDEX`, `FORCE INDEX`), `STRAIGHT_JOIN`, and query rewriting. After any upgrade to 3.x, expect regressions: run `ANALYZE TABLE` on all major tables, review critical plans with `EXPLAIN ANALYZE`, create histograms for skewed columns, and have optimizer hints ready. Aurora MySQL 3.x also gains descending indexes (eliminating `filesort` for `ORDER BY ... DESC`), functional indexes on expressions, invisible indexes for safe removal testing, CTE support, window functions, and skip scan — which allows efficient index usage even when the leading column of a composite index is not constrained [^497^]. After upgrade, queries that previously used workarounds may get better plans, but queries that relied on specific 5.7 optimizer behavior may regress.

Aurora MySQL 2.x had a custom hash join implementation for parallel queries; 3.x adopts the community MySQL 8.0 native hash join [^497^]. The query cache, present in 2.x, was removed in 3.x — workloads that relied on it must migrate to application-level caching (Redis/Memcached) or ProxySQL query cache, which showed a 5.2x improvement over uncached execution in AWS testing [^497^].

## 9.3 Performance Insights and Wait Event Analysis

### Average Active Sessions: The Primary Database Health Metric

Performance Insights centers on **Average Active Sessions (AAS)**: the number of sessions actively executing or waiting at any point [^457^]. The dashboard overlays AAS against maximum vCPU count — when AAS consistently exceeds vCPU count, queries queue and latency degrades non-linearly. The load chart breaks AAS by wait event: CPU (green), IO (orange), Lock (red), and Network (purple). A healthy database shows AAS below the vCPU line with a CPU-dominant mix.

**Important transition notice**: Performance Insights reaches end of life on June 30, 2026, replaced by CloudWatch Database Insights [^481^]. Clusters default to Standard mode (7-day retention, no execution plans, no on-demand analysis). Advanced mode (~$9/vCPU/month) provides 15-month retention, query plans, per-query statistics, and OS process analysis [^481^] [^482^]. For production Aurora MySQL, Advanced mode is effectively mandatory — Standard mode lacks the query-level detail needed to diagnose the bottlenecks below.

### Top Aurora-Specific Wait Events

Performance Insights breaks load down by wait events — the reasons sessions wait rather than execute. The following table lists the most operationally significant events for Aurora MySQL.

| Wait Event | Category | Root Cause | Remediation |
|:---|:---|:---|:---|
| `io/aurora_redo_log_flush` | I/O | Commit thread saturation from high-frequency small transactions; the dedicated commit thread cannot batch flushes fast enough [^59^] | Batch writes into larger transactions; review commit frequency; consider larger instance |
| `io/table/sql/handler` | I/O | Missing indexes, large table scans, or cold buffer pool causing storage I/O amplification [^8^] | Add covering indexes; check `EXPLAIN` for `ALL` scans; monitor `BufferCacheHitRatio` |
| `synch/mutex/innodb/buf_pool_mutex` | Lock | Buffer pool contention from excessive concurrent access [^8^] | Scale instance for larger buffer pool; reduce concurrency via connection pooling |
| `wait/lock/table/sql/handler` | Lock | Table-level lock waits from long-running transactions or DDL [^59^] | Keep transactions < 5 seconds; schedule DDL in maintenance windows; use `pt-online-schema-change` |
| `cpu` | CPU | Complex sorts, hash aggregations, or excessive row processing from missing indexes [^8^] | Add indexes; optimize queries to eliminate `filesort`; scale instance class |
| `io/file/innodb/innodb_data_file` | I/O | Data file reads bypassing buffer pool [^497^] | Add indexes; increase buffer pool; kill long-running scans |

### The Async Commit Paradox Revisited

In Chapter 7, we introduced the async commit paradox: Aurora's documentation claims worker threads never block on commit, yet `io/aurora_redo_log_flush` routinely appears as a top wait event. The resolution is that the flush work still happens — it just blocks the dedicated commit thread rather than worker threads.

When that commit thread becomes saturated, the wait manifests as `io/aurora_redo_log_flush` in Performance Insights. The remediation is batching writes (reducing commit frequency), not adding CPU, because the bottleneck is the commit thread's batching efficiency, not worker thread capacity [^59^]. This is why a workload of 10,000 single-row `INSERT` commits per second will show this wait event heavily, while the same throughput batched into 100-row multi-value inserts may show none at all.

For `io/table/sql/handler`, correlate with `BufferCacheHitRatio`. If the ratio is below 95% and this wait dominates, the working set exceeds buffer pool capacity — the fix is usually scaling to a larger instance (more RAM) rather than query optimization [^497^]. This connects directly to Chapter 3's buffer pool analysis: a cold pool drives storage reads, and storage reads on Aurora cross the network to the distributed storage tier.

Table-level lock waits deserve special attention in Aurora because DDL on the writer forcefully kills long-running reader transactions to acquire metadata locks [^59^]. When `ALTER TABLE` executes on the writer, readers holding metadata locks are terminated so the DDL can proceed — a behavior that terminates analytical reports or ETL jobs on readers without warning. All DDL must run in maintenance windows, and `pt-online-schema-change` should be preferred for large table alterations. This is the query performance layer of the locking behavior we covered in Chapter 6.

## 9.4 Common Anti-Patterns

### SELECT *, Implicit Conversions, Functions on Indexes, N+1 Queries

`SELECT *` prevents covering indexes (forcing base table access), increases network transfer, and can cause on-disk temporary tables. The fix is explicit column selection.

Implicit type conversions invalidate indexes silently. Comparing a string literal to an integer column forces a full table scan:

```sql
-- Bad: string literal vs INT column. EXPLAIN: type: ALL, rows: 1000000
SELECT * FROM orders WHERE user_id = '12345';
-- Good: types match. EXPLAIN: type: ref, key: idx_user_id, rows: 3
SELECT * FROM orders WHERE user_id = 12345;
```

Functions on indexed columns are the most common cause of index avoidance [^8^]. A B-Tree is sorted on original values, not transformed:

```sql
-- Bad: YEAR() prevents index usage. EXPLAIN: type: ALL
SELECT * FROM orders WHERE YEAR(created_at) = 2024;
-- Good: range scan. EXPLAIN: type: range, key: idx_created_at
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
-- Alternative: functional index (MySQL 8.0.13+)
ALTER TABLE orders ADD INDEX idx_created_year ((YEAR(created_at)));
```

N+1 queries execute one query for a list, then one per entity. At scale this generates hundreds of round-trips per request. Replace with a single `JOIN` or bulk `IN()` fetch:

```sql
-- N+1 pattern: 1 query for 100 customers, then 100 queries for orders = 101 queries
-- Fix: single JOIN, 1 query regardless of customer count
SELECT c.id, c.name, o.id AS order_id, o.total
FROM customers c LEFT JOIN orders o ON o.customer_id = c.id
WHERE c.id BETWEEN 1 AND 100;
```

### Pagination with Large OFFSET

`LIMIT offset, count` with large offsets is O(offset + count) because MySQL scans and discards all offset rows. `SELECT * FROM orders ORDER BY created_at LIMIT 10 OFFSET 100000` touches 100,010 rows to return 10.

**Keyset pagination** replaces the offset with a range condition, achieving O(log n + count) regardless of depth:

```sql
-- First page
SELECT * FROM orders ORDER BY id LIMIT 10;
-- Next page: cursor on last seen id. EXPLAIN: type: range, key: PRIMARY, rows: 10
SELECT * FROM orders WHERE id > 10 ORDER BY id LIMIT 10;
```

Keyset pagination requires an unambiguous ordering column and does not support arbitrary page jumps — acceptable for infinite-scroll but not for numbered pagination. For page-numbered UIs, the **deferred join** uses a covering index in the inner query to find IDs, then fetches full rows only for matches:

```sql
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders ORDER BY created_at LIMIT 10 OFFSET 100000
) ids USING (id);
```

The inner query touches only the `(created_at, id)` index, avoiding base table access for the 100,000 discarded rows. The outer query fetches full rows only for the 10 returned IDs. This reduces I/O by orders of magnitude when the offset is large [^8^]. The key constraint is that the inner query's `ORDER BY` column must be indexed for the optimization to work — otherwise the inner query itself becomes a full sort on disk.

### Missing Index Detection

The Performance Schema provides production-safe query analysis. `events_statements_summary_by_digest` aggregates statistics by query pattern, revealing high-scan queries:

```sql
-- Find queries scanning far more rows than returned
SELECT
    DIGEST_TEXT AS query_pattern,
    COUNT_STAR AS exec_count,
    SUM_ROWS_EXAMINED AS total_rows_examined,
    SUM_ROWS_SENT AS total_rows_sent,
    ROUND(SUM_ROWS_EXAMINED / NULLIF(SUM_ROWS_SENT, 0), 2) AS examined_per_sent
FROM performance_schema.events_statements_summary_by_digest
WHERE SUM_ROWS_EXAMINED > SUM_ROWS_SENT * 100
ORDER BY examined_per_sent DESC LIMIT 10;
```

An `examined_per_sent` ratio above 1,000 signals missing indexes or suboptimal access paths. A query with ratio 50,000 that returns 10 rows is scanning 500,000 rows per execution — on a 1 million row table, this almost certainly indicates a full table scan where an index should apply. Correlate with `EXPLAIN` to confirm `ALL` or `index` where `ref` or `range` is expected. For slow query log analysis, `pt-query-digest` ranks patterns by impact:

```bash
pt-query-digest --filter '$event->{Rows_examined} / $event->{Rows_sent} > 1000' \
    --type=slowlog /var/lib/mysql/slow.log
```

The `sys.statements_with_runtimes_in_95th_percentile` view provides quick outlier identification:

```sql
SELECT query, db, exec_count, total_latency, rows_examined_avg
FROM sys.statements_with_runtimes_in_95th_percentile
ORDER BY total_latency DESC LIMIT 10;
```

CloudWatch Database Insights Advanced mode provides per-query statistics and execution plans over 15 months, enabling detection of gradual plan regressions as data distributions shift [^481^]. Before the Performance Insights EOL deadline, validate that your missing-index workflow functions in Advanced mode — Standard mode lacks the query-level detail required for this analysis [^482^].

---

We now understand Aurora from storage to queries. Every layer — the distributed storage volume, the page structure, the buffer pool, the transaction system with its MVCC and undo logs, the lock manager, the redo log pipeline, the replication layer, and finally the query optimizer — is connected. A slow query may trace back to a cold buffer pool driving storage reads. A replication lag spike may trace back to a long-running reader blocking purge. A write throughput ceiling may trace back to the async commit thread saturating on single-row commits.

But knowledge isn't enough — you need to run it. Part IV moves from understanding to operating: monitoring, planning, tuning, and integrating everything into a production strategy that survives real workloads, real failures, and real 3 AM pages.
